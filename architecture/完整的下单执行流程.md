## 下单流程概览

- **入口**：应用侧拿到 `System` 句柄后，可以调用 `system.send_open_requests(...)` 发起买单/卖单。该方法会把订单封装成 `Command::SendOpenRequests` 并投递到引擎事件队列。  
  ```rust 133:141:barter/src/system/mod.rs
    pub fn send_open_requests(&self, requests: OneOrMany<OrderRequestOpen>)
    where
        Event: From<Command>,
    {
        self.send(Command::SendOpenRequests(requests))
    }
  ```

- **命令处理**：`Engine` 收到 `Command::SendOpenRequests` 后，进入 `Engine::action`，实际调用 `send_requests()` 将订单事件转成 `ExecutionRequest::Open`：  
  ```rust 205:236:barter/src/engine/mod.rs
            Command::SendOpenRequests(requests) => {
                info!(?requests, "Engine actioning user Command::SendOpenRequests");
                let output = self.send_requests(requests.clone());
                self.state.record_in_flight_opens(&output.sent);
                ActionOutput::OpenOrders(output)
            }
  ```

- **策略链路（可选）**：如果启用算法策略，在 `Engine::process` 内部会周期性调用 `generate_algo_orders()`。策略实现 `AlgoStrategy::generate_algo_orders` 产出 `OrderRequestOpen`，风险管理 `RiskManager::check` 审核，随后仍由 `send_requests()` 发送。[`46:67:barter/src/engine/action/generate_algo_orders.rs`]

## 发送与执行细节

1. **执行通道**：`send_requests()` 会按 `ExecutionTxMap` 查找目标交易所对应的 `Tx` 并发送 `ExecutionRequest::Open`：  
   ```rust 54:118:barter/src/engine/action/send_requests.rs
        self.execution_txs
            .find(&request.key.exchange)?
            .send(ExecutionRequest::from(request.clone()))
   ```

2. **执行管理器**：`ExecutionManager::run` 循环读取这些执行请求，借助索引映射成交易所原生订单参数，然后交给具体的 `ExecutionClient::open_order()` 执行；同时追踪超时并把成交/失败结果重新发回引擎状态：  
   ```rust 238:341:barter/src/execution/manager.rs
            request = self.request_stream.next() => match request {
                Some(ExecutionRequest::Open(request)) => {
                    let client_request = self.indexer.order_request(&request)?;
                    in_flight_opens.push(RequestFuture::new(
                        self.client.open_order(client_request),
                        self.request_timeout,
                        request,
                    ))
                }
            },
        response_open = next_open_response => { ... self.response_tx.send(event) ... }
   ```

3. **执行客户端**：真正触达交易所的逻辑由 `ExecutionClient` trait 定义（如 `open_order`, `cancel_order` 等）。仓库自带了 Mock 实现；要打通实盘，需要针对具体交易所（Binance、OKX 等）编写对应的 `ExecutionClient`。  
    ```rust 24:99:barter-execution/src/client/mod.rs
    pub trait ExecutionClient
    where
        Self: Clone,
    {
        const EXCHANGE: ExchangeId;
        type Config: Clone;
        type AccountStream: Stream<Item = UnindexedAccountEvent>;
        fn open_order(
            &self,
            request: OrderRequestOpen<ExchangeId, &InstrumentNameExchange>,
        ) -> impl Future<Output = Option<Order<...>>> + Send;
        ...
    }
    ```

4. **反馈循环**：`ExecutionManager` 把交易所回来的订单状态包装成 `AccountStreamEvent`，送回引擎，更新 `EngineState` 中的在途订单/持仓，供后续策略和风控使用。[`350:410:barter/src/execution/manager.rs`]

## 实际使用步骤

1. 使用 `SystemBuilder` 构建 `System`，确保 `ExecutionBuilder` 里为目标交易所添加了对应的执行客户端（Mock 或真实实现）。
2. 通过 `system.send_open_requests(...)` 直接下单；或在自定义策略里实现 `AlgoStrategy` 自动产生订单。
3. 回调/状态更新可在引擎 audit、账户流中捕获，或通过 `engine_state` 查询。

总结：下单入口位于 `System::send_open_requests`（命令式）或策略实现 `AlgoStrategy::generate_algo_orders`（自动化）。两条路径最终都汇聚到 `Engine::send_requests`，再由 `ExecutionManager` 调用具体 `ExecutionClient::open_order` 完成与交易所的交互。

## 下单额度在哪里定义？
下单“额度”（买入/卖出数量和价格）是在构造 `OrderRequestOpen` 时确定的。  
核心位置：

- `RequestOpen` 结构体声明于  
  ```rust 15:39:barter-execution/src/order/request.rs
  pub struct RequestOpen {
      pub side: Side,
      pub price: Decimal,
      pub quantity: Decimal,      // 订单数量就在这里定义
      pub kind: OrderKind,
      pub time_in_force: TimeInForce,
  }
  ```  
  也就是说，每个下单请求都内嵌一个 `RequestOpen`，其中的 `quantity`（以及 `price`）就是订单额度。

- 如果通过 `System::send_open_requests` 手动下单，需要在传入的 `OrderRequestOpen` 里自行设置 `state.quantity`。  

- 如果依赖策略自动生成订单，例如平仓逻辑，会在策略代码里填充额度：  
```rust 102:129:barter/src/strategy/close_positions.rs
  quantity: position.quantity_abs,  // 根据持仓决定下单数量
  kind: OrderKind::Market,
```
  这类策略/风控可以自定义数量来源、类型等。

- 风险管理器 `RiskManager::check`（`barter/src/risk/mod.rs`）可以对 `OrderRequestOpen` 进行过滤或调整，从而限制实际发送出去的额度。

额外约束（如最小下单量、数量步进）可通过 `InstrumentSpec`（`barter-instrument` 模块）维护，但最终真正的下单数值还是由生成 `OrderRequestOpen` 时的 `state.quantity` 决定。
